# Forensic Replay

Forensic replay in this repository is a safe, synthetic-only process for reproducing behavior observed in Comptext/Daimler experiments without preserving or redistributing sensitive data.

## Goals

- Recreate failure shapes, latency patterns, and analysis edge cases with synthetic fixtures.
- Preserve enough context for future agents to understand the issue and validate fixes.
- Feed sanitized, actionable findings back into `ProfRandom92/Comptextv7`.
- Avoid committing real Daimler payloads, proprietary customer identifiers, raw logs, secrets, API keys, bearer tokens, cookies, or credentials.

## Synthetic-only replay policy

All replay inputs must be synthetic. A replay fixture may mirror the structure of an observed issue, but field values must be invented and clearly marked as synthetic. Use placeholder case identifiers such as `synthetic-case-0001`, fake diagnostic codes, and generic descriptions that do not identify real vehicles, customers, employees, dealers, suppliers, or production systems.

Never paste raw production requests, raw response bodies, real stack traces containing customer context, or exported telemetry into this repository. If a real incident inspired the replay, document only the sanitized symptom pattern and the synthetic reproduction steps.

## Safe payload handling

Before committing replay assets:

1. Replace all customer, account, vehicle, employee, dealer, supplier, and system identifiers with synthetic values.
2. Remove secrets, API keys, bearer tokens, cookies, session identifiers, and credentials.
3. Run the fixture sanitizer:

   ```bash
   python scripts/sanitize_fixtures.py
   ```

4. Review `docs/reports/sanitization-report.md` and resolve suspicious findings before requesting review.
5. Keep replay reports concise and avoid raw payload dumps even when the payload is synthetic.

## Feeding findings back into Comptextv7

When replay identifies a likely runtime issue, create a sanitized handoff for `ProfRandom92/Comptextv7` that includes:

- The synthetic fixture or synthetic input shape.
- Expected behavior and observed behavior.
- Benchmark or replay report links from this repository.
- Any relevant p50, p95, p99, RPS, or error-rate changes.
- A statement that no real Daimler data is required to reproduce the issue.

The main application repository should receive code fixes and runtime changes. This experiment repository should retain the synthetic evidence, replay workflow, and regression context.

## What must never be committed

Do not commit:

- Real Daimler payloads or raw production logs.
- API keys, tokens, passwords, cookies, session IDs, or private certificates.
- Proprietary customer, employee, dealer, supplier, account, or vehicle identifiers.
- Data copied from live systems, customer environments, browser sessions, or private support channels.
- Reports that include raw sensitive payloads, even if the report is generated by a tool.

If sensitive data is accidentally staged, stop immediately, remove it, rotate the exposed credential if applicable, and ask a maintainer whether repository history cleanup is required.
